7.2 LLM Hoạt Động Như Thế Nào?

Cốt lõi của LLM rất đơn giản: dự đoán từ tiếp theo (next token prediction).

# LLM dự đoán từng token một:
Input:  "Thủ đô của Việt Nam là"
Step 1: "Thủ đô của Việt Nam là" → "Hà"      (xác suất 95%)
Step 2: "Thủ đô của Việt Nam là Hà" → "Nội"   (xác suất 99%)
Step 3: "Thủ đô của Việt Nam là Hà Nội" → "." (xác suất 80%)
sequenceDiagram participant U as 👤 Người dùng participant CC as 🖥️ Claude Code participant API as ☁️ Anthropic API participant LLM as 🧠 Claude (LLM) U->>CC: "Tạo hàm sort mảng" CC->>API: Gửi prompt + context API->>LLM: Tokenize & xử lý loop Sinh từng token LLM->>LLM: Dự đoán token tiếp theo end LLM->>API: Chuỗi tokens hoàn chỉnh API->>CC: Response (code) CC->>U: function sortArray(arr) { ... }

Luồng xử lý: Người dùng → Claude Code → API → LLM → Kết quả

📌

LLM không "hiểu" theo nghĩa con người. Nó tìm pattern thống kê từ hàng tỷ văn bản đã huấn luyện để sinh ra câu trả lời có xác suất cao nhất.

7.3 Token — Đơn Vị Xử Lý

LLM không đọc "từ" hay "ký tự" — nó đọc token. Token là đơn vị nhỏ nhất mà model xử lý.

Ví dụ tokenization

# Tiếng Anh — hiệu quả, ít token
"Hello world"       → ["Hello", " world"]              → 2 tokens
"programming"       → ["program", "ming"]               → 2 tokens

# Tiếng Việt — tốn nhiều token hơn
"Xin chào thế giới" → ["X", "in", " ch", "ào", " thế", " giới"] → 6 tokens
"lập trình"         → ["l", "ập", " tr", "ình"]        → 4 tokens

# Code — khá hiệu quả
"function main() {" → ["function", " main", "()", " {"] → 4 tokens
"console.log('hi')" → ["console", ".log", "('", "hi", "')"] → 5 tokens

Tại sao token quan trọng?

Yếu tốẢnh hưởngGhi chú
💰 Chi phíTính tiền theo số tokenInput tokens rẻ hơn output tokens
📏 Giới hạnMỗi model có context window cố địnhVượt giới hạn = mất thông tin đầu cuộc hội thoại
⚡ Tốc độNhiều token = chậm hơnOutput sinh ~50-100 tokens/giây
🌏 Ngôn ngữTiếng Việt tốn ~2-3x token so với tiếng AnhCùng ý nghĩa nhưng tốn nhiều token hơn
💡

Mẹo tiết kiệm token: Viết prompt bằng tiếng Anh khi có thể, dùng code block thay vì mô tả dài dòng, cung cấp context chính xác thay vì copy cả file.

7.5 Prompt — Cách Giao Tiếp Với AI

Prompt là câu lệnh / yêu cầu bạn gửi cho AI. Chất lượng prompt quyết định chất lượng kết quả.

Prompt tệ vs Prompt tốt

❌ Prompt tệ

Làm cho tôi cái web đẹp đẹp

Fix bug đi

Code không chạy, sửa giúp

Viết hàm xử lý data

✅ Prompt tốt

Tạo landing page responsive với header
có navbar, hero section với CTA button,
dùng Tailwind CSS, mobile-first.

Hàm fetchUsers() ở src/api.js trả về
undefined thay vì mảng users. Error ở
console: "Cannot read property map of
undefined". Hãy debug và fix.

Viết hàm TypeScript nhận mảng Product[],
lọc theo category, sort theo price giảm
dần, trả về top 10 kết quả.

4 Nguyên tắc viết prompt hiệu quả

#Nguyên tắcGiải thíchVí dụ
1Cụ thể (Specific)Nói rõ bạn muốn gì, format nào, ngôn ngữ nào"Viết hàm Python nhận list số nguyên, trả về list các số chẵn"
2Cung cấp ContextCho AI biết bối cảnh: project gì, file nào, error gì"Trong project Next.js 14, file app/page.tsx, lỗi hydration mismatch"
3Cho ví dụ (Few-shot)Đưa input/output mẫu để AI hiểu pattern"Input: [3,1,2] → Output: [1,2,3]. Viết hàm tương tự cho objects"
4Chia nhỏ (Decompose)Task lớn → nhiều bước nhỏ"Bước 1: Tạo schema DB. Bước 2: Viết API endpoints. Bước 3: Viết tests"

Các Prompt Pattern hữu ích

🎭 Role Pattern — Giao vai trò

# Cho AI một "nhân cách" chuyên môn
Bạn là senior TypeScript developer với 10 năm kinh nghiệm.
Review đoạn code sau và chỉ ra các vấn đề về:
- Type safety
- Error handling
- Performance

📝 Step-by-step Pattern — Yêu cầu từng bước

# Buộc AI suy nghĩ có cấu trúc
Hãy thiết kế REST API cho blog platform:
1. Liệt kê các entities cần thiết
2. Định nghĩa relationships giữa chúng
3. Viết API endpoints (method + path + body)
4. Viết TypeScript interfaces cho mỗi entity

🚧 Constraint Pattern — Đặt ràng buộc

# Giới hạn phạm vi output
Viết hàm sort KHÔNG dùng Array.sort() built-in.
Chỉ dùng vanilla JavaScript, KHÔNG dùng lodash.
Tối đa 20 dòng code.
Phải handle edge case: mảng rỗng, 1 phần tử.

7.7 Khái Niệm Nâng Cao

🌡️ Temperature — Độ "sáng tạo"

TemperatureHành viPhù hợp cho
0.0Luôn chọn token có xác suất cao nhất → kết quả ổn định, nhất quánCode, math, phân tích dữ liệu
0.5Cân bằng giữa chính xác và đa dạngViết email, tóm tắt, Q&A
1.0Chọn token ngẫu nhiên hơn → kết quả sáng tạo, bất ngờViết truyện, brainstorm, thơ
📌

Claude Code dùng temperature thấp cho code generation để đảm bảo kết quả chính xác và nhất quán.

🔧 System Prompt

System prompt là chỉ thị ẩn được đặt trước cuộc hội thoại, định nghĩa "tính cách" và quy tắc của AI.

# Ví dụ system prompt (người dùng không thấy)
{
  "role": "system",
  "content": "Bạn là trợ lý lập trình. Luôn trả lời bằng code kèm giải thích.
              Dùng TypeScript. Follow best practices. Không giải thích dài dòng."
}

# Trong Claude Code, CLAUDE.md chính là system prompt của bạn!
# File CLAUDE.md ở root project → Claude đọc đầu tiên

🚨 Hallucination — "Ảo giác" của AI

⚠️

Hallucination là khi AI tự tin nói sai — bịa ra thông tin không có thật nhưng nghe rất thuyết phục. Đây là hạn chế lớn nhất của LLM.

⚠️ AI hay hallucinate khi

• Bịa tên package không tồn tại
• Đưa ra API/method không có thật
• Tạo URL/link giả
• Trích dẫn "nguồn" không tồn tại
• Tự tin trả lời sai về phiên bản

✅ Cách phòng tránh

• Luôn verify thông tin AI đưa ra
• Check package trên npm/pypi trước khi cài
• Đọc official docs, không tin AI 100%
• Chạy test sau khi AI viết code
• Hỏi AI: "Bạn có chắc không?"

📚 RAG — Retrieval-Augmented Generation

RAG giúp AI trả lời chính xác hơn bằng cách tìm kiếm thông tin thực trước khi trả lời, thay vì chỉ dựa vào "trí nhớ" (training data).

7.9 Mẹo Sử Dụng AI Hiệu Quả

✅ NÊN LÀM (DO)

✓ Viết CLAUDE.md chi tiết cho project
✓ Cung cấp context đầy đủ trong prompt
✓ Chia task lớn thành nhiều bước nhỏ
✓ Review code AI sinh ra trước khi commit
✓ Chạy test sau mỗi thay đổi
✓ Bắt đầu conversation mới cho task mới
✓ Dùng /model để chọn model phù hợp
✓ Đọc error message và paste vào prompt
✓ Cho AI ví dụ input/output mong muốn
✓ Dùng tiếng Anh cho prompt kỹ thuật

❌ KHÔNG NÊN (DON'T)

✗ Tin AI 100% — luôn verify
✗ Copy-paste code mù quáng
✗ Dùng prompt mơ hồ: "làm cho đẹp"
✗ Bỏ qua error, nhờ AI fix mà không đọc
✗ Dùng Opus cho task đơn giản (tốn tiền)
✗ Conversation dài vô tận (quên context)
✗ Delete code không hiểu mà AI suggest
✗ Cài package AI gợi ý mà không check
✗ Bỏ qua viết test vì "AI code chắc đúng"
✗ Dựa hoàn toàn vào AI, không học
💡

Quy tắc vàng: AI là cộng tác viên, không phải phép màu. Bạn vẫn cần hiểu code, review kết quả, và ra quyết định cuối cùng.
