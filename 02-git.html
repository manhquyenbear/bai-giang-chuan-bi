🔀 Git Cơ Bản

Quản lý phiên bản mã nguồn — kỹ năng bắt buộc cho mọi developer

2.1 Git Là Gì?

Git là hệ thống quản lý phiên bản phân tán (Distributed Version Control System). Nó theo dõi mọi thay đổi trong mã nguồn, cho phép bạn quay lại bất kỳ thời điểm nào và cộng tác với nhiều người.

Lịch sử phiên bản Git — mỗi commit là một "ảnh chụp" toàn bộ project, trỏ ngược về commit trước đó

💡

Claude Code dùng Git để theo dõi mọi thay đổi nó tạo ra. Nếu không biết Git, bạn không thể review, undo, hoặc quản lý code mà Claude viết.

2.2 Cài Đặt Git

Cài đặt trên mỗi OS

🪟 Windows

# Tải và cài từ trang chủ
https://git-scm.com/download/win

# Hoặc dùng winget (Windows 10+)
$ winget install Git.Git

🍎 macOS

# Cài qua Homebrew (khuyến nghị)
$ brew install git

# Hoặc cài Xcode Command Line Tools
$ xcode-select --install
# Kiểm tra đã cài thành công
$ git --version
git version 2.44.0

Cấu hình lần đầu (BẮT BUỘC)

# Tên và email — xuất hiện trong mỗi commit
$ git config --global user.name "Nguyen Van A"
$ git config --global user.email "nguyenvana@gmail.com"

# Đặt branch mặc định là "main" (thay vì "master")
$ git config --global init.defaultBranch main

# Kiểm tra cấu hình
$ git config --list
user.name=Nguyen Van A
user.email=nguyenvana@gmail.com
init.defaultBranch=main
📌

--global áp dụng cho tất cả project. Muốn cấu hình riêng cho 1 project, bỏ --global và chạy trong thư mục project đó.

2.3 Ba Vùng Trong Git

Đây là khái niệm quan trọng nhất cần hiểu. Git chia thư mục project thành 3 vùng:

Ba vùng trong Git — Working Directory → Staging Area → Repository (.git directory)

VùngMô tảTrạng thái file
Working DirectoryThư mục bạn đang làm việc, chứa file thực tếModified (đã sửa, chưa add)
Staging AreaVùng "sân khấu" — chọn những thay đổi sẽ vào commit tiếpStaged (đã add, chờ commit)
Repository (.git)Kho lưu trữ trong thư mục ẩn .git/Committed (đã lưu vĩnh viễn)
💡

Tại sao cần Staging Area? Bạn sửa 10 file nhưng chỉ muốn commit 3 file liên quan đến feature A → git add chỉ 3 file đó. Giúp commit gọn gàng, có ý nghĩa.

2.4 Workflow Cơ Bản

Quy trình làm việc: sửa code → add → commit

Khởi tạo Git repo

# Tạo project mới và khởi tạo Git
$ mkdir my-website && cd my-website
$ git init
Initialized empty Git repository in /Users/qudomanh/my-website/.git/

# Hoặc clone repo có sẵn từ GitHub
$ git clone https://github.com/user/repo.git

git status — Xem trạng thái

$ git status
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        index.html
        style.css

nothing added to commit but untracked files present
💡

Luôn chạy git status trước khi add/commit để biết chính xác đang có những thay đổi gì. Đây là lệnh bạn sẽ dùng nhiều nhất!

git add — Đưa vào Staging

# Add từng file cụ thể
$ git add index.html
$ git add style.css

# Add tất cả file đã thay đổi
$ git add .

# Add theo pattern
$ git add *.js
$ git add src/
💡

git add . add TẤT CẢ — cẩn thận không add nhầm file nhạy cảm (.env, node_modules/). Nên dùng .gitignore (mục 2.8).

git commit — Lưu phiên bản

# Commit với message ngắn gọn
$ git commit -m "Thêm trang chủ với HTML và CSS"

# Commit kèm mô tả chi tiết (mở editor)
$ git commit

# Add + commit gộp (chỉ cho file ĐÃ TRACKED)
$ git commit -am "Sửa lỗi responsive header"

Viết commit message tốt

❌ Message tệ

"fix"
"update"
"abc"
"asdfsdf"
"done"

✅ Message tốt

"Thêm form đăng nhập"
"Sửa lỗi validate email"
"Xóa code thừa ở sidebar"
"Cập nhật API endpoint v2"
"Thêm unit test cho UserService"

2.5 Git Log & Diff — Xem Lịch Sử

git log — Xem lịch sử commit

# Log đầy đủ
$ git log
commit a1b2c3d (HEAD -> main)
Author: Nguyen Van A <nguyenvana@gmail.com>
Date:   Mon Apr 14 10:30:00 2026

    Thêm form đăng nhập

commit e4f5g6h
Author: Nguyen Van A <nguyenvana@gmail.com>
Date:   Sun Apr 13 09:00:00 2026

    Khởi tạo project

# Log ngắn gọn 1 dòng
$ git log --oneline
a1b2c3d (HEAD -> main) Thêm form đăng nhập
e4f5g6h Khởi tạo project

# Log dạng đồ thị (rất hữu ích khi có branch)
$ git log --oneline --graph --all
* f7g8h9i (feature/login) Xử lý auth
* d4e5f6g Tạo form login
| * a1b2c3d (HEAD -> main) Fix bug header
|/
* e4f5g6h Khởi tạo project

git diff — Xem thay đổi chi tiết

# Xem thay đổi chưa add (Working Dir vs Staging)
$ git diff
- <h1>Hello</h1>
+ <h1>Xin Chào</h1>

# Xem thay đổi đã add (Staging vs Repository)
$ git diff --staged

# Xem thay đổi của 1 file cụ thể
$ git diff index.html

# So sánh 2 commit
$ git diff a1b2c3d e4f5g6h
📌

Mẹo: Dùng git log --oneline để tìm mã commit, rồi git diff để xem thay đổi giữa 2 commit. Rất hữu ích khi debug "lúc nào code bị hỏng?".

Quay lại phiên bản trước

# Discard thay đổi chưa add (CẨN THẬN — mất code!)
$ git checkout -- index.html

# Bỏ file khỏi staging (giữ thay đổi trong Working Dir)
$ git reset HEAD index.html

# Quay lại commit trước (giữ code, bỏ commit)
$ git reset --soft HEAD~1

# Quay lại commit trước (bỏ cả code — CẨN THẬN!)
$ git reset --hard HEAD~1
⚠️

git reset --hard xóa vĩnh viễn mọi thay đổi chưa commit. Chỉ dùng khi bạn chắc chắn muốn bỏ hết!

2.6 Branch — Phân Nhánh

Branch cho phép bạn tạo "nhánh" độc lập để phát triển tính năng mới mà không ảnh hưởng code chính.

Branching — mỗi tính năng được phát triển trên branch riêng, lịch sử phân nhánh rồi merge lại

Các lệnh Branch

# Xem danh sách branch (* = branch hiện tại)
$ git branch
* main
  feature/login
  feature/dashboard

# Tạo branch mới
$ git branch feature/login

# Chuyển sang branch khác
$ git checkout feature/login

# Tạo + chuyển branch (1 lệnh gọn)
$ git checkout -b feature/signup

# Xóa branch (sau khi đã merge)
$ git branch -d feature/login

Merge — Gộp nhánh

# Bước 1: Chuyển về branch đích (thường là main)
$ git checkout main

# Bước 2: Merge branch feature vào
$ git merge feature/login
Merge made by the 'ort' strategy.
 src/login.js  | 45 +++++++++++++++++++++
 src/auth.js   | 23 +++++++++++
 2 files changed, 68 insertions(+)

Xử lý Merge Conflict

Khi 2 branch sửa cùng dòng trong cùng file, Git không tự merge được → conflict:

# Git báo conflict
$ git merge feature/login
CONFLICT (content): Merge conflict in src/header.js
Automatic merge failed; fix conflicts and then commit.

# File bị conflict sẽ có dạng:
<<<<<<< HEAD
<h1>Welcome</h1>
=======
<h1>Xin Chào</h1>
>>>>>>> feature/login

# Bước 1: Mở file, chọn phần code đúng, xóa dấu <<< === >>>
# Bước 2: Add và commit
$ git add src/header.js
$ git commit -m "Resolve merge conflict ở header"
💡

Giảm conflict: Pull thường xuyên, chia nhỏ branch, mỗi người sửa phần khác nhau. Commit nhỏ và thường xuyên!

2.7 GitHub (Remote Repository)

GitHub là nền tảng lưu trữ Git repo trên cloud, cho phép cộng tác và chia sẻ code.

GitHub — cầu nối giữa các developer

git clone — Tải repo về

# Clone repo từ GitHub (lần đầu)
$ git clone https://github.com/username/my-project.git
$ cd my-project

# Clone vào thư mục tùy chỉnh
$ git clone https://github.com/username/my-project.git ten-thu-muc

git remote — Quản lý kết nối

# Xem remote hiện tại
$ git remote -v
origin  https://github.com/username/my-project.git (fetch)
origin  https://github.com/username/my-project.git (push)

# Thêm remote (nếu tạo repo local trước)
$ git remote add origin https://github.com/username/my-project.git

git push — Đẩy code lên

# Push lần đầu (set upstream)
$ git push -u origin main

# Các lần sau chỉ cần
$ git push

# Push branch feature
$ git push -u origin feature/login

git pull — Kéo code về

# Kéo thay đổi mới nhất từ GitHub
$ git pull

# Kéo từ branch cụ thể
$ git pull origin main
💡

Quy trình cộng tác: git pull trước khi code → Viết code → git add → git commit → git pull (kiểm tra conflict) → git push.

2.8 .gitignore — Bỏ Qua File

File .gitignore cho Git biết không theo dõi những file/thư mục nào.

Ví dụ .gitignore cho dự án Node.js

# Dependencies — hàng nghìn file, không cần lưu vào Git
node_modules/

# Biến môi trường — TUYỆT ĐỐI không push lên GitHub!
.env
.env.local
.env.production

# Build output — tạo lại được từ source
dist/
build/
.next/

# Log files
*.log
npm-debug.log*

# OS files — rác hệ điều hành
.DS_Store
Thumbs.db

# IDE config
.vscode/
.idea/

# Secrets & keys
*.pem
*.key
⚠️

KHÔNG BAO GIỜ commit file .env hoặc API key lên GitHub! Dù repo private, key vẫn nằm trong lịch sử Git mãi mãi. Nếu lỡ commit, phải đổi key ngay.

📌

Thứ tự: Tạo .gitignore trước khi git add. Nếu đã add nhầm node_modules/, chạy: git rm -r --cached node_modules/ rồi commit lại.

Workflow hoàn chỉnh: Tạo project mới

# 1. Tạo project
$ mkdir my-app && cd my-app

# 2. Khởi tạo Git
$ git init

# 3. Tạo .gitignore TRƯỚC
$ echo "node_modules/" > .gitignore
$ echo ".env" >> .gitignore

# 4. Tạo file code
$ touch index.js README.md
$ npm init -y

# 5. Commit đầu tiên
$ git add .
$ git status          # Kiểm tra — không có node_modules!
$ git commit -m "Khởi tạo project my-app"

# 6. Kết nối GitHub và push
$ git remote add origin https://github.com/user/my-app.git
$ git push -u origin main
